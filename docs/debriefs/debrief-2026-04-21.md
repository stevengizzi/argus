## Findings

### F-01 — Log-volume regression: `pattern_strategy.py:298`
- **Severity:** CRITICAL (not data-integrity, but operational — a 182 MB single-session log materially slows every post-market debrief, consumes disk, and on long-horizon runs will pressure the host filesystem)
- **Safety tag:** `safe-during-trading`
- **Mechanism:** `pattern_strategy.py:298` emits `logger.info("%s: evaluating %s with partial history (%d/%d)", ...)` on every symbol × strategy × candle once the per-symbol candle deque is ≥50% full of `min_detection_bars`. With 30 registered strategies × 6,366 viable symbols × N candles during warm-up, this produces 782,273 INFO lines per session (84.4% of the log, ~150 MB of text).
- **Why it's safe to fix:** `record_evaluation()` two lines later already writes the same event into `evaluation.db` with `reduced_confidence: True` metadata. The `logger.info` adds no information the DB doesn't already capture.
- **Fix:** Downgrade to `logger.debug`, or remove entirely.
- **In any Phase 3 prompt?** No. New finding.

### F-02 — `getattr(pos, "avg_entry_price", 0.0)` reads wrong field
- **Severity:** CRITICAL — cause of "+4,222,124,650,659.50R" ETHA trade and "+5,519,117,189.5R Avg R" on the Dashboard today
- **Safety tag:** `weekend-only`
- **Mechanism:** `Position` dataclass has a field called `entry_price`, not `avg_entry_price`. The `getattr(pos, "avg_entry_price", 0.0)` calls in `_create_reco_position` and `_reconstruct_known_position` silently return 0.0. Every downstream R-multiple, unrealized P&L, MFE/MAE calculation then divides a real market price by a near-zero risk-per-share (`entry - stop` where entry is 0), producing values in the trillions.
- **In any Phase 3 prompt?** **Yes — FIX-04 P1-C1-C01 (CRITICAL).** Nothing new to do here; just verify the fix handles all call sites.

### F-03 — Bracket amendment leaks positions
- **Severity:** CRITICAL — silently opens IBKR positions the UI never shows and the Order Manager never tracks
- **Safety tag:** `weekend-only`
- **Mechanism:** When an entry fill has slippage > $0.01, `_handle_entry_fill` invokes the DEC-366 bracket-amendment logic which cancels and re-places the stop and T1 at delta-shifted prices. The correlation between the 45 untracked broker positions and the bracket-amendment log line is **45/46 = 98%**. The two cases where the log shows an "orphan T1 fill" (`T1 fill for LCID but no matching position`, `T1 fill for ETHA but no matching position`) are the clearest direct evidence: a bracket leg fired at IBKR after ARGUS had already marked the position closed. For KBR specifically (533-share ORB entry at 13:40:32), the log shows clean T1 + T2 fills totaling 533 SELLs, yet IBKR reported KBR: -200 at EOD — meaning extra SELL fills happened that ARGUS's state machine did not see or expect.
- **Plausible code path (hypothesis, not fully verified):** The amendment cancels and re-submits stop + T1 but **does not touch T2**. The original T2 at the pre-slippage price remains active. In cases where the original T2 is now below the fill price (which it always is for a BUY slippage — original T2 was computed pre-slip, actual entry fill is above it), IBKR treats the stale T2 as a marketable LIMIT and fills it immediately. Result: extra SELL shares hit IBKR that ARGUS's Order Manager state machine doesn't reconcile against the known bracket.
- **Why F-02's fix doesn't resolve this:** F-02 is about *reading back* `entry_price` incorrectly during reconciliation. F-03 is about *forward order management* producing excess SELLs. Different code paths, different root cause. They overlap only in that some of the `entry_price=0` reconstructions in F-02 may have been triggered *by* F-03 orphan fills.
- **In any Phase 3 prompt?** **No.** Needs a dedicated session or a scope expansion on FIX-04.

### F-04 — Flatten-retry loop against non-existent positions
- **Severity:** HIGH — infinite retries, contributes to XNDU symptom, generates the 2,011 short-sale rejections
- **Safety tag:** `weekend-only`
- **Mechanism:** Once a position is leaked (F-03) or an entry is reconstructed with `entry_price=0` (F-02), the Order Manager may list it in `_managed_positions` and `_broker_confirmed` but IBKR may have no record of it. Every time-stop or trail-stop or EOD-flatten trigger sends a SELL to IBKR. With no underlying long, IBKR interprets this as a short sale. For shortable symbols it succeeds (creating the short positions Steven saw: ETHA, LBRDK, KBR, TQQQ, NOK). For hard-to-borrow symbols it rejects with error 201 "not available for short sale". DEC-369 "broker-confirmed positions never auto-closed" protects the internal state from being corrected — so the retry loop runs indefinitely until EOD.
- **XNDU specifically:** 40+ consecutive flatten attempts between 15:50 and 15:55 UTC after Steven's manual close. Even after shutdown trigger at 15:51:05, the loop kept firing SELL orders until the data-feed subscription was torn down at 16:01.
- **Fix options:** (a) Invalidate `_broker_confirmed[symbol]` when IBKR's portfolio snapshot excludes the symbol for N consecutive checks (e.g., 3 × 60s = 3 minutes); (b) Treat IBKR error 201 on a flatten as *definitive evidence* the position doesn't exist and remove from `_managed_positions`; (c) both.
- **In any Phase 3 prompt?** **No.** New finding.

### F-05 — Debunked: "duplicate trade_id" is a log-truncation artifact
- **Severity:** COSMETIC (downgraded from initial CRITICAL)
- **Safety tag:** `safe-during-trading`
- **Mechanism:** `trade_logger.py:113` uses `trade.id[:8]` in the log message format. ULIDs encode a millisecond timestamp in their first 10 characters; `[:8]` truncates to roughly a 1-second window. 10 trades closing within the same second legitimately share the first 8 ULID characters. The full 26-char ULID is unique per trade and is what gets written to the `trades.id` PRIMARY KEY column. No data integrity issue.
- **Fix:** Change `trade.id[:8]` to `trade.id[:12]` (or log the full ULID) for debug readability.
- **In any Phase 3 prompt?** No, but trivially safe to add.
- **RESOLVED IMPROMPTU-07 (2026-04-23):** `trade_logger.py:126` (`log_trade`) + `:648` (`log_orchestrator_decision`) both changed from `[:8]` to `[:12]`. Regression: `tests/analytics/test_trade_logger.py::TestF05LogTruncationWidth` (2 tests asserting the 12-char prefix is present in the emitted log line).

### F-06 — Shadow Trades UI: MFE/MAE unit mismatch
- **Severity:** MEDIUM — user-visible data display bug
- **Safety tag:** `safe-during-trading`
- **Mechanism:** Backend `CounterfactualTracker._process_bar` (counterfactual.py:487-494) stores `max_favorable_excursion` and `max_adverse_excursion` as **dollar amounts** (`bar_high - entry_price`, `entry_price - bar_low`). Frontend `ShadowTradesTab.tsx:477,480` passes those values to `RMultipleCell`, which formats them as `${value.toFixed(2)}R`. A favorable excursion of $0.004 displays as `+0.00R`.
- **Fix:** Either (a) convert to R-multiples in the REST response (divide by `entry_price - stop_price`), or (b) swap `RMultipleCell` for a `DollarCell` in the MFE/MAE columns. (a) is preferable because the columns are currently labeled as R.
- **In any Phase 3 prompt?** No. FIX-12 (frontend) has findings but none on the MFE/MAE unit mismatch specifically.
- **RESOLVED IMPROMPTU-07 (2026-04-23):** Option (a) taken. `_enrich_with_r_multiples()` in `argus/api/routes/counterfactual.py` computes `mfe_r` / `mae_r` fields at serialization time (`excursion / |entry − stop|`, MAE flipped to negative). Original `max_favorable_excursion` / `max_adverse_excursion` dollar fields preserved for backward-compat. `ShadowTrade` TypeScript interface extended; `ShadowTradesTab.tsx` column cells + sort keys now bind `mfe_r`/`mae_r`. Regression: `tests/api/test_counterfactual_api.py::TestF06MfeMaeRMultiples` (2 tests — field presence + known-value math), `argus/ui/src/features/trades/ShadowTradesTab.test.tsx` (1 new test — `test_f06_mfe_mae_r_columns_render_from_r_multiple_fields`). Schema unchanged.
